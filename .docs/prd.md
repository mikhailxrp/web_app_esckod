## О продукте

**Название:** Код доступа: корпорация

**Что это:** Веб-приложение, цифровая часть гибридной настольной детективной игры. Игрок выступает детективом, расследующим гибель Виктора Пака. Расследование разбито на сцены, в каждой — игровая мини-миссия (взлом сайта по типу Wordle, расшифровка по Плейферу/Виженеру, удалённый доступ через пазл-трубопровод и симуляцию Windows). Между миссиями — скриптовые чаты с персонажами и изучение «удалённых» файлов. Доступ к приложению — только по уникальному ключу из физической коробки с игрой.

**Целевая аудитория:** Покупатели физической настольной игры (тираж 3000 коробок).

**Деплой:** VPS Beget + Nginx + PM2 + GitHub Actions. БД и хранилище файлов — в инфраструктуре Beget (РФ).

**Бюджет:** 300k ₽ (50/50). Срок: 2-3 месяца.

## Стек

См. `CLAUDE.md` раздел «Tech Stack» — там полный стек с обоснованиями.

Кратко: Next.js 16 (App Router) + TypeScript + Prisma + PostgreSQL + Auth.js v5 + Tailwind + Zod + Zustand. REST API Routes (не Server Actions). Файлы — в Beget Cloud Storage (S3-совместимое API).

## Функциональные требования (верхний уровень)

Подробности каждого модуля — в `.docs/modules/`.

### 1. Авторизация и доступ

Регистрация только по уникальному ключу из коробки (1 ключ → до 5 пользователей). Пароль генерируется системой и отправляется на email. Сессия 24 часа. Изолированные роли: PLAYER и ADMIN — отдельные таблицы и эндпоинты входа. Защита от enumeration attack на reset-password. Rate limiting на всех публичных эндпоинтах.

→ Детали: `.docs/modules/auth.md`

### 2. Онбординг

Одноразовая экскурсия по интерфейсу при первом входе. Реализуется как оверлей (react-joyride) поверх настоящего dashboard, подсвечивает реальные элементы по `data-onboarding-id`. Skip отсутствует — только последовательное «Далее».

→ Детали: `.docs/modules/onboarding.md`

### 3. Блокировка устройств с малым экраном

Полноэкранная статичная заглушка с текстом «Игра доступна только на десктопе. Откройте приложение на компьютере.». Показывается на любом устройстве с экраном меньше `1024 × 700 px` или с touch-указателем при недостаточной ширине. Реализуется через гибридную детекцию: сервер (User-Agent + Client Hints) + клиент (viewport + pointer + maxTouchPoints).

→ Детали: `.docs/modules/mobile-block.md`

### 4. Главный игровой экран

Статусбар (STATUS / TARGET / ACCESS), **три плашки миссий** (Взлом сайта / Дешифратор / Удалённый доступ), запускаемые через формы ввода полей, чат Детектива, чат Марины (скрыт до триггера), история операций, кнопки «Подсказки от Детектива», «Начать заново», «Финальный отчёт» (доступна по двойному триггеру).

**Прогресс по конкретным слотам не отображается на плашке** — игрок видит только три кнопки запуска. Статус прохождений доступен только через историю операций (OperationLog).

**Если все слоты определённого типа отключены** (`isActive=false`) — соответствующая плашка скрывается на dashboard (не показывается вовсе). Это исключительная ситуация, защищённая серверным инвариантом (см. «Ключевые архитектурные принципы», принцип 7).

### 5. Чаты (граф диалога)

Диалоги хранятся как граф: реплики (`ChatScript`) + переходы (`ChatTransition`). Три типа условий перехода: ALWAYS / CHOICE / TRIGGER. Поддерживает линейные участки, ветвления (выбор игрока), ожидания серверных событий. Аудио-сообщения (как голосовые в Telegram) с кнопкой «Показать расшифровку» для слабослышащих. Финальный выбор Марины фиксируется автоматически в `ChatState.finalChoice`.

→ Детали: `.docs/modules/chats.md`

### 6. Подсказки Детектива

Отдельная функция (не часть чата). Кнопка-иконка на главном экране, открывает модалку с подсказками по сюжету. Подсказки показываются по одной, переход — только по «Далее» (защита от спойлеров). Управляются админом, сбрасываются при перезапуске игры.

→ Детали: `.docs/modules/hints.md`

### 7. История операций (Logs)

Постоянно видна внизу dashboard. Записи по строгим шаблонам (`constants/logTemplates.ts`). Два уровня: технические (с подстановкой URL/паролей/IP) и обзорные («Миссия X — пройдена»). Пишет ТОЛЬКО сервер через `lib/operationLog.ts`. Клиент только читает.

### 8. Миссия Crack (Взлом сайта)

Wordle-механика: 25-30 слов по 5 букв на сетке, цветовая индикация позиций. Повторяемая по слотам (`MissionSlot` с `missionType=CRACK`). Минимальный набор: `CRACK_P2`, `CRACK_VUZ`. Параметры (`crackMaxAttempts`) настраиваются админом per-slot. Состояние сессии — в `CrackSession` на сервере. При провале — пересоздание поля, бесконечные попытки.

**Запуск через форму URL + login.** Сервер ищет активный слот по точному совпадению `targetUrl + targetEmail`. При несовпадении — лог ошибки доступа (`crack_launch_failed`), попытки запуска не ограничены.

**Кнопка «Пропустить»** появляется после 2 исчерпаний всех попыток Wordle подряд. При пропуске миссия считается пройденной — те же логи и триггеры, что при честном прохождении.

→ Детали: `.docs/modules/missions-crack.md`

### 9. Миссия Decipher (Дешифратор)

Два типа шифра: Плейфер (таблица 6×6 русских букв) и Виженер (числа-позиции над буквами). Повторяемая по слотам. Минимум: `DECIPHER_SHANTAZH`, `DECIPHER_MARKOVA`. Лимита попыток нет. Защита `/complete` через флаг `MissionProgress.metadata.lastAttemptCorrect`.

**Запуск через форму пути к папке.** Сервер ищет активный слот по точному совпадению `folderPath`. При несовпадении — лог ошибки (`decipher_launch_failed`), попытки запуска не ограничены.

**Кнопка «Пропустить»** появляется после 2 неправильных вводов расшифрованного слова подряд. Счётчик сбрасывается при правильном вводе.

→ Детали: `.docs/modules/missions-decipher.md`

### 10. Миссия RDP (Удалённый доступ)

Два сценария: упрощённый (6×6, без таймера) и увеличенный (7×7, два пути, таймер, бесконечные попытки). Игрок вводит IP вслепую — сервер находит слот по IP. Симуляция Windows: папки и файлы (PDF) с возможностью разблокировки папки паролем из Decipher. Сюжетные триггеры срабатывают **автоматически на сервере** после того, как игрок просмотрел все доступные файлы (через `/file-viewed`):
- Сценарий 1: лог «два активных сеанса + новый IP» + симуляция закрывается. **RDP-окно сценария 1 нельзя закрыть** до разблокировки хотя бы одной запароленной папки — можно только свернуть в иконку.
- Сценарий 2: активация чата Марины (`marinaTriggered=true`)

**Пароли и пути разблокированных папок** сохраняются в `OperationLog` (лог `rdp_folder_unlocked`) — игроку не нужно снова открывать RDP-окно для повторного просмотра.

**Кнопка «Пропустить»** в сценарии 2 появляется после 2 истечений таймера. В сценарии 1 пропуск недоступен.

**Миссия доступна только на десктопе.**

→ Детали: `.docs/modules/missions-rdp.md`

### 11. Финальный отчёт

Доступен по двойному триггеру: все активные миссии пройдены + игрок сделал финальный выбор в чате Марины. Вопросы с radio-кнопками. Серверная проверка ответов. Текст финала зависит от `ChatState.finalChoice` (`PROTECT`/`ACCUSE` — два варианта, расширяемо). Сдаётся один раз, повторно — только просмотр результата.

→ Детали: `.docs/modules/final-report.md`

### 12. Перезапуск игры (Restart)

Игрок может в любой момент полностью сбросить прогресс. Серверная транзакция: DELETE `MissionProgress`, `CrackSession`, `OperationLog`, `UserHintProgress`; UPDATE `ChatState`, `GameProgress`. НЕ сбрасывается: `User` (включая `onboardingDone`), `AccessKey.currentActivations`. Rate limit 3/мин.

→ Детали: `.docs/modules/restart.md`

### 13. Админ-панель

Изолированный логин (`/admin-login`). Управление: ключами (CRUD + CSV-импорт + лимит активаций), пользователями (бан/разбан, выгрузка email), администраторами (CRUD с защитой последнего/себя), слотами миссий (визард создания, редактирование, мягкое отключение), чатами (граф: реплики + переходы + валидация связности), файлами RDP (привязка к слотам, isLocked для папок), финальным отчётом (вопросы + контенты концовок), подсказками Детектива, глобальными настройками (`AppSettings`).

→ Детали: `.docs/modules/admin.md` (общий обзор) + специфичные модули.

### 14. Глобальные настройки приложения (`AppSettings`)

Singleton-таблица. Хранит: `defaultMarketingConsent` (дефолт галки маркетинга в форме регистрации, по умолчанию `false` по 152-ФЗ/GDPR), `supportEmail` (email техподдержки в сообщениях об ошибках регистрации), `privacyPolicyUrl` (ссылка на политику обработки данных).

→ Детали: `.docs/modules/app-settings.md`

## Что НЕ делаем (Out of Scope)

**Контент:**

- Тексты диалогов и реплик — предоставляет заказчик
- Аудиофайлы чатов — предоставляет заказчик
- PDF-документы для RDP — предоставляет заказчик
- Дизайн и графика — предоставляет заказчик
- Тексты вопросов и финалов — предоставляет заказчик

**Функционал:**

- **Email-рассылки и редактирование текста писем** — out of scope (закрыто с заказчиком). Resend используется только для системных писем (генерация пароля, reset). Тексты системных писем — захардкожены в коде.
- Мобильное приложение (iOS/Android)
- Мультиязычность
- Интеграция с платёжными системами
- Аналитика и дашборд статистики
- Поддержка нескольких игр одновременно (закладываем архитектуру через слоты, но не реализуем вторую игру)
- Real-time чаты (WebSocket) — все чаты скриптовые
- OAuth (Google/GitHub) — только credentials через Auth.js v5

**Сторонние сервисы:**

- Настройка и поддержка сторонних сайтов, к которым выдаются пароли
- SEO-оптимизация
- Нагрузочное тестирование

## Поддерживаемые браузеры и устройства

**Браузеры:** Chrome, Firefox, Safari (включая iOS), Edge — последние 2 мажорные версии каждого. Старые версии официально не поддерживаются.

**Устройства:**

- **Десктоп** (Windows, macOS, Linux): все миссии и функции при разрешении экрана от `1024 × 700 px`.
- **Планшеты в горизонтальной ориентации** с подключённой мышкой/клавиатурой (≥ 1024 px ширины): поддерживаются.
- **Смартфоны, планшеты в портретной ориентации, любые экраны < 1024 × 700 px**: показывается полноэкранная заглушка «Игра доступна только на десктопе. Откройте приложение на компьютере.». Игровой функционал недоступен. См. `.docs/modules/mobile-block.md`.

**Мобильная адаптация UI** — out of scope (заказчик подтвердил). UI рассчитан только на десктопные разрешения.

## Правки и поддержка

**В рамках бюджета:**

- До 2 итераций правок после сдачи каждого этапа
- Правки в рамках утверждённого scope
- 2 месяца гарантийной поддержки с момента приёмки финального этапа (бесплатное исправление багов в реализованном функционале)

**Не входит в бюджет:**

- Доработки, новые фичи, изменения дизайна
- Изменения сторонних сервисов (Beget, Resend, обновления PostgreSQL)
- Восстановление данных при потере (бэкапы — на стороне Beget)
- Адаптация под новые версии браузеров после сдачи
- Любые изменения контента (делается заказчиком через админку)

## Ключевые архитектурные принципы

1. **Слоты миссий вместо хардкода.** Каждое прохождение миссии = запись в `MissionSlot`. «Виктор», «Марина», «P2 Digital», «ВУЗ» — это значения в БД, не имена в коде. Новый слот добавляется через админку без изменений кода.

2. **Граф диалога вместо линейной последовательности.** Реплики (`ChatScript`) + переходы (`ChatTransition`) с условиями ALWAYS/CHOICE/TRIGGER. Поддерживает любые сюжеты: линейные, с ветвлениями, с ожиданием событий.

3. **Серверная защита всех критичных переходов.** Прогресс, логи, состояние чата — пишутся только сервером изнутри игровых эндпоинтов при реальных событиях. Клиент НЕ имеет публичного API на запись этих данных. Защита `/complete` через флаги в `MissionProgress.metadata` (`lastAttemptCorrect`, `puzzleSolved`).

4. **Управляемая сложность.** Параметры мини-игр (количество попыток Crack, размер пазла RDP, длительность таймера) — в `MissionSlot`, не в коде. Заказчик тонко настраивает кривую сложности через админку без релизов.

5. **Изоляция админа от игрока.** Отдельная таблица `AdminUser`, отдельный credentials-провайдер Auth.js v5, отдельная ветка в `proxy.ts`. Компрометация игрового аккаунта не даёт доступа к админке.

6. **Контент отделён от прогресса.** Глобальные таблицы (`MissionSlot`, `ChatScript`, `ChatTransition`, `RdpFile`, `FinalReportQuestion`, `FinalReportContent`, `DetectiveHint`, `AppSettings`) — управляются админом и НЕ сбрасываются при перезапуске игры. Прогресс игрока — в персональных таблицах с каскадным удалением при удалении `User`.

7. **Двухуровневая структура миссий: 3 плашки UI + 6 слотов БД.** UI группирует слоты по `MissionType` (CRACK / DECIPHER / RDP) в одну плашку — игрок видит 3 кнопки запуска миссий. Сервер при запуске ищет конкретный слот по полям формы (URL+login, folderPath, IP). Это даёт ощущение нелинейности, не ломая существующую модель слотов.

   > **Инвариант:** в системе всегда должен существовать минимум один активный слот каждого `MissionType`. Админка блокирует деактивацию/удаление последнего активного слота типа.

8. **Защита от параллельных действий (concurrency).** Один аккаунт = команда за столом, играющая с нескольких устройств. Критичные таблицы (`GameProgress`, `MissionProgress`, `CrackSession`, `ChatState`) защищены полем `version` (optimistic locking). Mutate-эндпоинты возвращают HTTP 409 при конфликте, клиент рефетчит данные с тостом «состояние изменилось». Restart защищён UI-таймером 5 сек + PostgreSQL advisory lock. См. `.docs/modules/concurrency.md`.

## Роадмап

Декомпозиция на фазы по принципу **vertical slices**: для каждой фичи, зависящей от контента, сначала админский CRUD, затем игровая логика. Это гарантирует, что каждая фича проверяема end-to-end сразу после завершения.

Фазы намеренно мелкие (1–3 дня). Внутри фазы — таски (1–2 дня работы, 3–7 файлов каждый) — расписываются агентом при инициации фазы через `.docs/prompts/phase-init.md`.

**Принцип:** каждая фаза заканчивается рабочим, проверяемым результатом. Админский CRUD контента всегда предшествует игровой логике, которая этот контент использует.

> **⚠️ О формулировке «Модель `X` в БД» ниже:** все Prisma-модели (18 шт.) и enum'ы созданы в **Фазе 0** одной init-миграцией. В последующих фазах модели **не создаются заново** — пункты вида «Модель `ChatState` в БД», «Модели `MissionSlot`, `MissionProgress` в БД» означают **наполнение (сидер) + API + UI** для уже существующей модели. Новая миграция нужна только при изменении полей модели; источник правды по схеме — `.docs/database.md`.

### Фаза 0 — Фундамент

**Цель:** Настроить инфраструктуру разработки. Готовый каркас Next.js + Prisma + БД + базовая защита роутов. Никакого UI пока.

Что должно работать в конце:

- `npm run dev` запускает приложение, БД подключена
- Prisma миграции работают, схема создана
- `proxy.ts` редиректит на /login для защищённых роутов
- Базовый layout с заглушкой
- Health-check эндпоинт отвечает

### Фаза 0.5 — Деплой и CI/CD

**Цель:** Приложение работает на VPS Beget по реальному домену с SSL ещё с пустым каркасом. Каждая последующая фаза просто докатывается на уже работающий прод.

Что должно работать:

- VPS настроен (Nginx, PM2, Node.js)
- SSL-сертификаты (Let's Encrypt через certbot)
- Beget Managed Database подключена
- Beget Cloud Storage подключён
- GitHub Actions: pipeline сборки и деплоя при пуше в `main`
- Smoke-tests на пустом каркасе (health-check, redirect /dashboard → /login)

### Фаза 1 — Авторизация (полный цикл)

**Цель:** Игрок может зарегистрироваться по ключу, получить пароль на email, войти, выйти, восстановить пароль. Админ может войти в админку.

Что должно работать:

- Регистрация по ключу (с проверкой `currentActivations < maxActivations`)
- Email с паролем (Resend)
- Вход + сессия 24 часа
- Reset password (с защитой от enumeration attack)
- Логин админа на отдельном `/admin-login`
- Rate limiting на всех публичных эндпоинтах
- Сидер: первый админ из ENV, заглушки `AppSettings`

**Архитектурные решения (admin auth):**

- Страницы входа и восстановления пароля администратора вынесены в отдельную route-группу `app/(auth-admin)/` со своим `layout.tsx`. Это позволяет полностью изолировать визуальный стиль admin-части (светлая тема, фоновое изображение, лого) от игрового `app/(auth)/` (тёмная хакерская тема). Route group не влияет на URL — `/admin-login` и `/admin-reset-password` работают как обычно.
- Восстановление пароля администратора доступно по `/admin-reset-password` (не `/reset-password`) — отдельный маршрут, чтобы избежать конфликта двух route-групп на одном URL и дать каждому потоку свой layout.
- Admin-специфичные design-токены (`admin.accent`, `admin.card-bg`, `admin.input-bg` и др.) добавлены в `tailwind.config.ts` в секцию `admin` — изолированы от game-токенов, легко расширяются при добавлении admin dashboard в Фазе 3.
- Глобальный webkit-autofill стиль в `globals.css` применял тёмный фон игры (`#0D1117`) ко всем инпутам. Для admin-форм добавлен override `.admin-input:-webkit-autofill` — класс вешается на инпуты admin-форм и перекрывает глобальное правило.
